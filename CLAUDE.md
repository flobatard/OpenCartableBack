# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Project

Backend FastAPI d'une plateforme pédagogique auto-hébergée (Raspberry Pi) : un prof compose des cours par blocs et les partage à ses élèves via des liens publics. Le cahier des charges complet (architecture cible, enjeux, modèle de données, roadmap J0→J5) est dans **Descriptions.md** — le lire avant tout travail de fond ; il fait foi et doit être mis à jour quand une décision d'architecture change. Le projet est en plein jalon J1 (contenu) : taxonomie des matières, classification des niveaux d'étude, modèle BDD des cours (blocs + bibliothèque de ressources, découplés), **API des cours** (`app/courses/` : CRUD cours, structure des blocs, édition du contenu texte/exercice/document) et **bibliothèque de ressources S3** (`app/core/storage.py` + `app/resources/` : CRUD complet + flow presigned — presign PUT → confirmation HEAD → ressource `disponible`, sans création de bloc) sont en place ; les liens publics et la présignature côté élève (J2) restent à livrer. L'utilisateur échange en français.

## Architecture

### Auth : resource server pur (décision structurante)
Le flow OIDC (Authorization Code + PKCE) est **entièrement géré par la SPA Angular**. L'API n'émet jamais de token et ne stocke aucune identité/mot de passe : elle valide le JWT Zitadel de chaque requête (signature RS256 via JWKS découvert depuis l'issuer, vérif issuer/audience/exp). Toute la logique IdP est **confinée dans `app/core/auth.py`** (exigence du cahier des charges : pouvoir changer d'IdP en ne touchant que ce module). Seuls réglages : `OIDC_ISSUER` et `OIDC_AUDIENCE`.

Sémantique des erreurs à préserver : pas de credentials ou token invalide → **401** + `WWW-Authenticate: Bearer` (jamais 403) ; IdP injoignable ou discovery/JWKS malformé → **503** ; jamais de 500, jamais le token dans les logs. Routes protégées = `Depends(get_current_user)` (`GET /api/v1/me` est l'exemple de référence) ; `/api/v1/health` reste public. Un second régime d'accès (token de partage opaque pour les élèves consultant sans compte) arrivera au jalon J2 : il aura sa **propre dépendance d'autorisation**, distincte de `get_current_user`.

### Comptes applicatifs & onboarding (`users`)
La feature `app/users/` persiste les comptes (profs **et/ou** élèves — rôles cumulables `est_prof`/`est_eleve`) : `GET /api/v1/users/me` **auto-provisionne** la ligne au premier appel (upsert `ON CONFLICT DO NOTHING` sur `sub`, seule donnée IdP persistée avec l'e-mail en snapshot) ; `PUT /api/v1/users/me/profile` valide et enregistre le profil (système scolaire ∈ `education_levels.systeme`, niveaux du même système, matières existantes) en **remplacement complet** — il sert aussi d'édition de profil. Les M2M `user_subjects`/`user_education_levels` sont qualifiées par `contexte` (« enseigne »/« apprend ») ; profil complet ⟺ `onboarded_at` non NULL. **Invariant préservé : `get_current_user` ne touche jamais la base** — la résolution `sub → users` vit dans `app/users/service.py` (`get_or_create_by_sub`), pas dans `app/core/auth.py`. L'ordre des `execute` des fonctions du service est un contrat des tests (fausse session FIFO, `tests/test_users_api.py`).

### Configuration en couches
Priorité décroissante : variables d'env > `.env` > `config/<APP_ENV>.yaml` (`APP_ENV` par défaut : `development`). `config/*.yaml` = valeurs publiques versionnées ; `.env` = secrets et overrides locaux uniquement. `DATABASE_URL` est **assemblée** dans `app/core/config.py` depuis les composants `POSTGRES_*` (mot de passe percent-encodé) ; une URL complète collée dans `DATABASE_URL` prend le pas et est normalisée pour asyncpg (`sslmode`/`channel_binding` traduits en `ssl=`). Réglages S3 sur le même principe : `S3_ENDPOINT_URL`/`S3_REGION`/`S3_BUCKET` + TTL/plafond sont publics (YAML), `S3_ACCESS_KEY`/`S3_SECRET_KEY` sont des secrets (`.env`) ; tous ont un défaut → le conftest n'a rien à poser. Attention : `settings = get_settings()` s'évalue **à l'import** — dans les tests, les variables d'env sont posées en tête de `tests/conftest.py` avant tout import de `app.*` ; conserver cet ordre.

### Découpage
- Stack 100 % async : SQLAlchemy 2.0 + asyncpg partout, y compris `alembic/env.py`. Ne pas introduire de driver/engine synchrone.
- Package-by-feature : chaque domaine (courses, subjects, resources…) = un package `app/<feature>/` avec `schemas.py`, `service.py` (logique métier), `router.py` (HTTP), monté dans `create_app()` sous `settings.API_V1_PREFIX`.
- Les modèles SQLAlchemy vivent dans **`app/models/<nom>.py`** (un module par modèle) et doivent être importés + listés dans `__all__` de `app/models/__init__.py`, sinon Alembic autogenerate ne les voit pas.

### Taxonomie des matières (`subjects`)
Table unique **auto-référencée** (`app/models/subject.py`) : discipline (profondeur 0) → domaine (1) → sous-domaine (2) → sujet (3), profondeur flexible (une branche peut s'arrêter avant 3, CHECK en base) ; unicité `(parent_id, nom)` en `NULLS NOT DISTINCT` (Postgres 15+), FK `parent_id` en `ondelete=CASCADE`. Chaque nœud porte un `code` = chemin slug complet unique (ex. `mathematiques.algebre.algebre-lineaire`) dont dérive son id **uuid5 déterministe**. La taxonomie pré-remplie (~475 nœuds, lycée → master) vit dans **`app/subjects/seed_data.py` — module de données pur et APPEND-ONLY** : ne jamais changer `SEED_NAMESPACE` ni renommer/supprimer un `code` existant (la migration de seed idempotente `ON CONFLICT DO NOTHING` et les IDs en dépendent) ; tout ajout passe par une nouvelle data migration réutilisant `iter_rows()`. Lecture : `GET /api/v1/subjects/tree` — une seule requête SQL, arbre assemblé en O(n) dans `app/subjects/service.py` (les relations ORM `parent`/`children` ne sont pas chargées : lazy-load async interdit). Le CRUD prof arrivera au J1.

### Classification des niveaux d'étude (`education_levels`)
Même motif que `subjects`, en plus contraint : table auto-référencée (`app/models/education_level.py`) à deux profondeurs — cycle (0, ex. « Collège ») > classe (1, ex. « 6e ») —, un arbre par système scolaire (`systeme`, plusieurs pays occidentaux pour l'instant), unicité `(systeme, parent_id, nom)` en NULLS NOT DISTINCT. Chaque nœud porte les **pivots internationaux** `cite` (CITE/ISCED 2011, NULL si le nœud couvre plusieurs niveaux, ex. « Supérieur ») et `age_min`/`age_max` : c'est par eux que des cours se rapprocheront entre pays — les `nom` sont des noms propres nationaux, jamais traduits. Différence clé avec subjects : les `code` sont **écrits à la main** et préfixés système (ex. `fr.college.6e`), jamais dérivés du nom affiché. Le seed (22 nœuds, primaire → doctorat, voie générale) vit dans **`app/education_levels/seed_data.py`** — module de données pur et APPEND-ONLY, `SEED_NAMESPACE` propre et figé (distinct de celui des subjects) ; tout ajout (maternelle, voie pro, BTS/CPGE, arbre étranger) passe par une nouvelle data migration réutilisant `iter_rows()`. Lecture : `GET /api/v1/education-levels/tree` (même motif flat select + arbre O(n)). Pas de relations ORM ni de CRUD prévu ; le lien cours ↔ niveaux est implémenté (M2M `course_education_levels`, voir ci-dessous) ; le profil utilisateur (niveaux/matières par contexte) est implémenté via `app/users/`.

### Modèle des cours (`courses`, `blocks`, `resources`)
Modèles dans `app/models/{course,block,resource}.py`, API dans `app/courses/` (section suivante). Un cours appartient à un user (`owner_id`, CASCADE) et est classé par deux M2M **sans qualificatif** `course_subjects`/`course_education_levels` (PK composite `course_id` en tête, index inverse pour les facettes de recherche — motif `user_subjects` sans `contexte`). Son contenu = **blocs ordonnés** : `position` SmallInteger **sans unicité `(course_id, position)`** (voulu : le réordonnancement réécrit les positions côté service ; tri stable `ORDER BY position, id`), `titre`/`description` **facultatifs et communs à tous les types** (métadonnées d'en-tête du bloc, distinctes du `content`), quatre types (CHECK `ck_blocks_type`) dont le `content` JSONB est un **contrat applicatif documenté dans la docstring de `block.py`** — `texte` (`{"markdown": ...}`, jamais de HTML brut ; formules LaTeX `$…$`/`$$…$$` admises dans la chaîne, rendues par KaTeX côté front ; les liens externes vivent dans le markdown, l'ancien type `lien` a été supprimé), `exercice` (`{"enonce": md, "questions": [{id, enonce md, type "texte_libre", reponse_attendue texte}]}` — `reponse_attendue` = corrigé du prof en texte simple, **jamais servi aux élèves** (le J2 filtrera le content des liens publics) ; les `questions[].id` uuid sont générés en service et **stables à vie** : les soumissions élèves (J2) et la review IA référenceront `(block_id, question_id)`, ne jamais les régénérer à l'édition), `document` (`{"legende", "affichage": inline|telechargement}` — pont vers une ressource de la bibliothèque, référencée par la **colonne** `resource_id`, jamais dans le content) et `module` (`{}`, placeholder — contrat défini au J4). CHECK de cohérence `ck_blocks_document_coherence` : **seuls les blocs `document` peuvent porter `resource_id`**, qui est **nullable** (un document naît vide et se remplit dans l'éditeur) avec FK **`ON DELETE CASCADE`** (supprimer une ressource supprime les blocs pointeurs — décision actée : un document sans son fichier n'a pas de sens). `resources` = **bibliothèque de fichiers S3 du cours, indépendante des blocs** : `s3_key` **plate** unique (`uuid/nom-original`), type ∈ document/image/audio/video (`module` retiré — les modules interactifs sont des blocs, leur schéma sera recréé au J4 ; l'ancienne table `modules`, restée orpheline, a été droppée), **`statut` `en_attente` → `disponible`** (la ligne est créée avant l'upload direct navigateur→S3, puis la confirmation HEAD S3 la passe à `disponible` — voir `app/resources/` ci-dessous). **Aucune relation ORM** sur ces modèles (lazy-load async interdit). Piège : autogenerate ne détecte pas la modification d'un `CheckConstraint` (il voit en revanche un changement de `ondelete` de FK) et ne génère jamais de migration de données — élargir `ck_blocks_type`/`ck_resources_type` = migration manuelle (la FK `blocks.resource_id`, créée sans nom, s'appelle `blocks_resource_id_fkey` en base). À venir : `share_links` (J2), `search_vector` FTS (J3).

### API des cours (`app/courses/`)
Toutes les routes résolvent d'abord `user = get_or_create_by_sub(db, auth)` puis scopent au propriétaire : un cours d'autrui est **introuvable (404, un seul select `id + owner_id`), jamais interdit (403)** — on ne divulgue pas son existence.
- `GET /api/v1/courses` (mes cours, tri `updated_at desc`, avec `subject_ids`/`education_level_ids`/`block_count`) ; `POST /courses` (201 — `subject_ids`/`education_level_ids` **optionnels**, dédoublonnés et validés par lookup, 422 « Matières inconnues »/« Niveaux d'étude inconnus » ; timestamps relus en RETURNING) ; `GET /courses/{id}` (détail **avec blocs** triés `position, id`).
- Blocs : `POST /courses/{id}/blocks` (201 — type ∈ `texte|exercice|document|module`, tous créables ici : un bloc `document` naît **vide** (`resource_id` NULL) et se remplit dans l'éditeur ; `titre`/`description` optionnels posés dès la création ; contenu par défaut par type conforme à `block.py`, position = max+1) ; `PUT /courses/{id}/blocks/order` (204 — la liste doit contenir **exactement** les blocs du cours, positions réécrites 0..n-1 en un executemany Core sur la Table ; à garder déclarée avant un futur `PUT /blocks/{block_id}`) ; `PATCH /courses/{id}/blocks/{block_id}` (`BlockUpdate` — édition **partielle** : `titre`/`description` sur **tous les types**, `content` = **union de formes disjointes** `TexteContent | ExerciceContent | DocumentContent` (smart union sans discriminant, `extra="forbid"` des deux côtés — `DocumentContent` en **dernier** dans l'union, tous ses champs ont un défaut donc `{}` valide en document) dont la forme doit correspondre au type du bloc (422 sinon, levée **avant toute écriture** ; `module` n'a aucune forme éditable avant le J4) — pour l'exercice, le service préserve les ids de questions fournis (qui doivent exister dans le bloc, 422 « Questions inconnues » sinon), génère un uuid4 `str` pour les nouvelles (un `uuid.UUID` n'est pas JSON-sérialisable par asyncpg) et applique une sémantique **remplacement** (question absente = supprimée) ; **`resource_id`** (blocs `document` uniquement, 422 sinon) pointe/détache la ressource : `null` explicite détache, un uuid est validé en service — ressource du **même cours** et `disponible`, 422 « Ressource inconnue »/« Ressource non disponible » (422 et non 404 : le bloc, lui, existe — motif « Matières inconnues ») ; seuls les champs présents dans le payload sont appliqués via `model_fields_set`, un `titre`/`description`/`resource_id` explicitement `null` l'efface ; au moins un champ requis (422 sinon) ; `extra="forbid"`, `content` remplacé par un **nouveau dict** car la mutation in-place du JSONB n'est pas détectée par l'ORM) ; `DELETE /courses/{id}/blocks/{block_id}` (204 — pas de renumérotation, trous tolérés ; **ne touche jamais ni `resources` ni S3** : la ressource pointée par un bloc document reste dans la bibliothèque).
- **Toute mutation de blocs bump `course.updated_at`** (écriture d'attribut sur l'instance chargée par le contrôle de propriété) pour que le cours remonte dans la liste.
- **Nettoyage S3 aux suppressions** : `delete_course` énumère les `s3_key` du cours (et `delete_resource`, côté `app/resources/`, relit la sienne), puis `storage.delete_many` les purge du bucket **après le commit** (le cascade DB ne touche pas S3) — arbitrage : un échec S3 après commit laisse un orphelin dans le bucket, jamais une réf DB pointant un objet absent ; le vrai filet (job de réconciliation des orphelins) reste à écrire.
- Tests `tests/test_courses_api.py` : fausse session FIFO **étendue** — les `Update` sont tracés sans consommer la file, un `Insert` porteur de RETURNING (celui de `courses`) la consomme pour servir les timestamps ; `get_or_create_by_sub` consomme toujours le premier `[user]` de la file et commite une fois. Le faux client S3 est injecté via `app.dependency_overrides[get_storage]` (helper `_FakeStorage`, enregistre les clés supprimées).

### Stockage S3 (`app/core/storage.py`) et bibliothèque de ressources (`app/resources/`)
Bucket **privé**, jamais exposé : l'API mint des URL présignées, l'upload est **direct navigateur→S3** (on ne fait pas transiter les binaires par le backend — contrainte Pi : RAM/BP). Toute la logique boto3 est **confinée dans `app/core/storage.py`** (même exigence de remplaçabilité que l'IdP : c'est le **seul** module autorisé à importer `boto3`), exposée en dépendance `get_storage` (overridable en test comme `get_db`). `presign_put`/`presign_get` sont du **calcul local** (signature, aucune I/O) → appel sync OK dans un handler async ; `head`/`delete_many` parlent au réseau → déportés dans un thread (`run_in_threadpool`) pour ne pas bloquer l'event loop.

`app/resources/` est le **CRUD de la bibliothèque du cours, indépendante des blocs** (une ressource existe sans bloc ; les blocs `document` la pointent via `PATCH .../blocks/{id}` côté `app/courses/`), scopé au propriétaire (404 jamais 403). Routes : `GET /api/v1/courses/{id}/resources` (liste, tri `created_at desc, id`, **`en_attente` incluses** — le front les affiche atténuées et permet de purger un upload raté ; jamais de `s3_key` dans `ResourceRead`, détail interne) ; le flow d'upload (Descriptions.md §5.2) — 1) `POST .../resources` (201) déclare le fichier (`nom_original`/`mime`/`taille`/`type ∈ document|image|audio|video` — **`module` exclu**, J4 ; `taille` ≤ `S3_MAX_UPLOAD_BYTES`), crée la ligne **`en_attente`** (clé plate `<resource_id>/<nom-sanitizé>`) et renvoie l'**URL présignée PUT** ; 2) le navigateur PUT le binaire sur cette URL ; 3) `POST .../resources/{rid}/confirm` (**200, sans body**) fait un **HEAD S3** — objet absent ou `ContentLength` ≠ `taille` déclarée → **409**, ressource déjà confirmée → 409 — et passe le statut à **`disponible`** (retour `ResourceRead` ; **ne crée AUCUN bloc**) — ; `PATCH .../resources/{rid}` (renommage : seul `nom_original` change, la **clé S3 reste figée**, le `Content-Disposition` des prochains downloads suit) ; `DELETE .../resources/{rid}` (204 — delete puis **purge S3 après commit**, motif `delete_course` ; les blocs `document` pointeurs partent avec elle par la FK `CASCADE`, aucun execute supplémentaire) ; `GET .../resources/{rid}/download` (**URL présignée GET**, TTL court — 409 tant que la ressource n'est pas `disponible`). Toute mutation (confirm/rename/delete) bump `course.updated_at`. Ordre des `execute` documenté par fonction (contrat FIFO de `tests/test_resources_api.py`). Les mutations d'une ressource chargée en ORM passent par **mutation d'attribut** (pas d'UPDATE Core explicite — flush au commit).

Dev local : **MinIO** dans `docker-compose.yml` (multi-arch ARM64, console `:9001`) + job one-shot `minio-createbucket`. **Caveat** : une URL présignée mintée par l'api-en-conteneur pointe `minio:9000`, **injoignable par le navigateur** ; pour tester l'upload de bout en bout, lancer **uvicorn en local** (`config/development.yaml` → `localhost:9000`) — c'est le workflow de dev déjà en place (uvicorn local + Postgres docker sur `5429`).

### Tests sans dépendances externes
`tests/conftest.py` fournit : clé RSA de session + `make_token(...)` (signe des JWT RS256 avec `kid` de test, iss/aud/exp valides par défaut et surchargables), `mock_jwks` (monkeypatch de `jwks_cache.get_signing_key` — aucun réseau), `client`. Pour tester une route protégée sans crypto : `app.dependency_overrides[get_current_user]` ; pour une route qui lit la base : `app.dependency_overrides[get_db]` avec une fausse session (motif de référence : `tests/test_subjects_api.py`). Aucun test ne requiert Postgres ni Zitadel.

## Décisions actées (ne pas "corriger")

- **Pas de pgvector** : la vectorisation n'est pas actée ; si elle se fait ce sera probablement ChromaDB (jalon J5). Ne pas proposer d'extension Postgres.
- **Pas de reverse proxy dans ce repo** : nginx est fourni et branché par l'infra ; le compose expose l'API sur le port 8000, c'est voulu.
- **Dépendances IA de `requirements.txt` (langchain*, sentence-transformers, chromadb, redis, psycopg2-binary) volontairement conservées** bien qu'inutilisées au J0 — ne pas les purger.
- **`scripts/` est écrit et maintenu à la main par l'utilisateur** — ne pas le modifier sans demande explicite.
- Contrainte transverse (Descriptions.md §5.8) : cible Raspberry Pi ARM64 — déporter le lourd (transferts via URL S3 présignées, jobs asynchrones), images Docker multi-arch.
- L'utilisateur s'occupe toujours de faire les commande alembic pour créer les migrations et les appliquer
